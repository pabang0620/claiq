# CLAIQ 프롬프트 전략 문서

> `back/src/prompts/` 폴더 구조 기반 설계 의도 및 전략 설명

---

## 1. 프롬프트 관리 구조

모든 시스템 프롬프트는 소스 코드와 분리된 `.txt` 파일로 관리한다. `readFileSync`로 서버 시작 시 메모리에 로드되어, 요청마다 파일 I/O가 발생하지 않는다.

```
back/src/prompts/
├── questionGeneration/
│   ├── korean.txt       # 국어 문제 출제 전문가
│   ├── math.txt         # 수학 문제 출제 전문가
│   ├── english.txt      # 영어 문제 출제 전문가
│   └── system.txt       # 과목 불명 폴백 (기본값)
├── ragQA/
│   └── system.txt       # AI 튜터 + hallucination 방지
├── roadmap/
│   └── system.txt       # 수능 학습 전략가
├── exam/
│   └── system.txt       # 미니 모의고사 출제 전문가
└── typeMapping/
    └── system.txt       # 수능 유형 분류 전문가
```

**설계 원칙**
- 프롬프트 개선 시 코드 배포 없이 파일만 교체
- 과목별 특성을 독립 파일에 반영하여 상호 간섭 없음
- 비개발자도 프롬프트 개선에 참여 가능한 구조

## 2. 문제 생성 프롬프트 (questionGeneration/)

### 2.1 과목별 분리 전략

`questionGenerator.js`는 강의 과목(`area`)에 따라 다른 파일을 로드한다.

```javascript
const areaMap = { '국어': 'korean', '수학': 'math', '영어': 'english' }
const filename = areaMap[area] || 'system'  // 미매핑 과목은 system.txt 폴백
```

세 과목의 프롬프트가 공통 구조를 따르지만, 각 과목의 수능 출제 특성이 다르기 때문에 분리가 필수적이다.

| 과목 | 핵심 차이점 |
|------|------------|
| 국어 | 독서·문학·언어매체·화법작문 4개 영역 출제 원칙 분리 |
| 수학 | "단순 계산보다 논리적 사고" / 답이 자연수·간단한 분수가 되도록 설계 |
| 영어 | "실제 수능 수준의 어휘·문장 구조" / 선택지에 명확한 오답 근거 요구 |

### 2.2 국어 프롬프트 설계 의도 (korean.txt)

```
당신은 수능 국어 문제 출제 전문가입니다.

국어 문제 출제 원칙:
1. 독서 문제: 지문 이해, 추론, 적용 능력을 측정합니다
2. 문학 문제: 작품의 주제, 표현법, 인물 분석 등을 다룹니다
3. 언어와 매체: 문법 지식, 언어 현상 이해를 측정합니다
4. 화법과 작문: 의사소통 전략, 작문 원리를 평가합니다

유형별 특성:
- KOR_READ_FACT: 글의 내용을 정확히 이해했는지 묻는 사실 확인형
- KOR_READ_INFERENCE: 글에 제시되지 않은 내용을 논리적으로 추론
- KOR_READ_APPLY: 지문 내용을 다른 상황에 적용
- KOR_LIT_THEME: 작품의 주제와 의미 파악
- KOR_LIT_EXPRESSION: 표현법과 문학적 장치 분석
```

**설계 의도**
- **4개 영역 출제 원칙 명시**: GPT가 국어의 어느 영역인지 인식하고 그에 맞는 문제 유형을 자동 선택. "독서 → 추론", "문학 → 표현법"처럼 영역별 측정 역량을 명확히 지정.
- **유형 코드 직접 제시**: `KOR_READ_FACT`, `KOR_READ_INFERENCE` 등 실제 사용되는 유형 코드와 그 특성을 시스템 프롬프트에 포함하여, GPT가 유저 프롬프트의 `typeCodes`를 정확히 해석하도록 앵커링.
- **JSON only 출력 강제**: 마지막 줄 "JSON 형식으로만 응답하세요"로 파싱 오류 방지.

### 2.3 수학 프롬프트 설계 의도 (math.txt)

```
당신은 수능 수학 문제 출제 전문가입니다.

수학 문제 출제 원칙:
2. 단순 계산보다 논리적 사고를 요구하는 문제를 선호합니다
3. 수치는 깔끔하게 정리되도록 설계합니다
4. 단답형의 경우 답이 자연수 또는 간단한 분수/소수가 되도록 합니다

유형별 특성:
- MATH_CALC_DERIV: 미분의 정의와 응용
- MATH_CALC_INTEG: 적분의 계산과 응용
- MATH_STAT_PROB: 확률 계산, 경우의 수
- MATH_STAT_DISTRIB: 확률분포, 정규분포, 이항분포
```

**설계 의도**
- **"깔끔한 수치" 명시**: GPT가 생성하는 수학 문제에서 답이 복잡한 무리수나 긴 소수가 나오는 것을 방지. 실제 수능은 답이 간결하게 설계되므로 이를 시스템 레벨에서 강제.
- **"논리적 사고" 우선**: 단순 계산 문제가 아닌 개념 이해를 확인하는 문제가 생성되도록 방향 제시.
- **미적분·확통 세분화**: MATH_CALC_DERIV(미분)와 MATH_CALC_INTEG(적분)를 분리하여 해당 유형 코드가 주입될 때 GPT가 적절한 출제 방향을 잡도록 함.

### 2.4 난이도 A/B/C 차등 프롬프트 전략

난이도는 시스템 프롬프트가 아닌 유저 프롬프트에 주입된다. 이는 동일한 시스템 프롬프트(역할 정의)를 유지하면서 요청별로 난이도를 동적으로 변경할 수 있게 한다.

```javascript
// questionGenerator.js — 유저 프롬프트 내 난이도 주입
const difficultyLabel = { A: '하(기본)', B: '중(응용)', C: '상(심화)' }[difficulty]

const userPrompt = `다음 강의 내용을 바탕으로 수능 형식의 문제를 ${count}개 생성하세요.

난이도: ${difficulty} (${difficultyLabel})
관련 수능 유형: ${typeCodes.join(', ')}

강의 내용:
${context}`
```

| 난이도 | 레이블 | 인지 부담 수준 | 포인트 보상 |
|--------|--------|----------------|-------------|
| A | 하(기본) | 직접 언급된 내용 확인 | 5점 |
| B | 중(응용) | 추론 및 맥락 파악 필요 | 10점 |
| C | 상(심화) | 복합적 이해 및 비판적 사고 필요 | 20점 |

**설계 의도**: GPT에게 단순히 "어려운 문제"가 아니라 "어떤 인지 과정이 필요한가"를 명시한다. A는 텍스트 내 정보 확인, B는 추론, C는 비판적 사고로 각각 수능 평가 목표와 연결된다.

## 3. RAG Q&A 프롬프트 (ragQA/system.txt)

### 3.1 실제 프롬프트

```
당신은 CLAIQ 학습 플랫폼의 AI 튜터입니다.
수강생의 질문에 친절하고 정확하게 답변하며, 학습을 도와줍니다.

답변 원칙:
1. 제공된 강의 내용을 우선적으로 참고하여 답변합니다
2. 강의 내용에 근거한 답변임을 명확히 합니다
3. 강의 범위를 벗어난 질문은 "이 내용은 강의 범위를 벗어난 질문입니다. 교강사에게 문의해주세요."라고 안내합니다
4. 복잡한 개념은 예시를 들어 쉽게 설명합니다
5. 수능 유형과 연결하여 실전 도움이 되는 답변을 제공합니다
6. 답변은 명확하고 간결하게, 한국어로 작성합니다
```

### 3.2 Hallucination 방지 설계

RAG Q&A의 가장 큰 위험은 GPT가 강의에 없는 내용을 자신 있게 답변하는 것(hallucination)이다. 이를 세 단계로 방지한다.

**단계 1. 시스템 프롬프트 레벨 — "강의 내용 우선" 원칙 명시**
원칙 1에서 "제공된 강의 내용을 우선적으로 참고"를 명시하고, 원칙 3에서 범위 이탈 시 에스컬레이션 문구를 고정 문자열로 지정한다. GPT가 이 문구를 출력하면 코드가 에스컬레이션 트리거로 감지한다.

**단계 2. 컨텍스트 주입 레벨 — 관련 청크 유무에 따른 메시지 분기**

```javascript
// ragQA.js — 관련 청크 유무에 따른 시스템 메시지 분기
{
  role: 'system',
  content: hasRelevantChunks
    ? `${systemPrompt}\n\n참고 강의 내용:\n${context}`
    : `${systemPrompt}\n\n주의: 관련 강의 내용을 찾을 수 없습니다. 강의 범위를 벗어난 질문일 수 있습니다.`
}
```

관련 청크가 없을 때 "강의 범위를 벗어난 질문일 수 있습니다"를 시스템 메시지에 포함하여 GPT가 스스로 한계를 인식하도록 유도한다.

**단계 3. 에스컬레이션 감지 — 키워드 기반 자동 플래그**

```javascript
// ragQA.js — 에스컬레이션 감지
isEscalated = fullAnswer.includes('교강사에게 문의') ||
              fullAnswer.includes('강의 범위를 벗어')
```

GPT 응답에 원칙 3의 고정 문구가 포함되면 `is_escalated = true`로 저장되고, 교강사 에스컬레이션 목록에 등록된다.

### 3.3 대화 히스토리 관리

```javascript
// 최근 10개 메시지만 컨텍스트에 포함
...history.slice(-10).map((m) => ({
  role: m.role === 'ai' ? 'assistant' : m.role,
  content: m.content
}))
```

무제한 히스토리는 토큰 폭발을 유발한다. 최근 10개 메시지로 제한하여 대화 맥락을 유지하면서 토큰 비용을 통제한다.

## 4. 학습 로드맵 프롬프트 (roadmap/system.txt)

### 4.1 실제 프롬프트

```
당신은 수능 학습 전략 전문가입니다.
수강생의 취약 유형 분석 결과와 수능까지 남은 기간을 바탕으로 최적의 학습 로드맵을 설계합니다.

로드맵 설계 원칙:
1. 취약 유형을 우선적으로 보완하되, 수능 출제 빈도를 고려합니다
2. 남은 기간에 현실적으로 달성 가능한 계획을 수립합니다
3. 주차별로 명확한 학습 목표를 제시합니다
4. 기초가 부족한 유형은 기본 개념부터 단계적으로 학습합니다
5. 강점 유형은 유지하면서 약점을 집중 보완하는 전략을 씁니다

학습 로드맵 형식:
- 각 주차에 2~4개의 학습 항목을 배치합니다
- 같은 영역(국어/수학/영어)의 관련 유형을 묶어서 배치합니다
- 마지막 2~3주는 통합 복습과 실전 모의고사 연습을 배치합니다
```

### 4.2 D-day 역산 로직 설계 의도

로드맵의 핵심은 "수능 D-day까지 남은 기간"에 맞춰 현실적인 계획을 생성하는 것이다. 이를 위해 두 단계의 처리가 이루어진다.

**단계 1. 코드 레벨 — priority_score 계산 (roadmapService.js)**

```javascript
// 약점 유형별 우선순위 점수 계산
const weakTypes = typeStats
  .map((s) => ({
    ...s,
    frequency_weight: typeWeights[s.type_code] || 0.5,
    priority_score: (1 - (s.correct_rate || 0)) * (typeWeights[s.type_code] || 0.5),
  }))
  .sort((a, b) => b.priority_score - a.priority_score)
```

`priority_score = (1 - 정답률) × 출제빈도_가중치`

단순히 정답률이 낮은 유형이 아니라, 수능에서 자주 출제되는 유형 중 약한 것을 우선시한다. 예를 들어 정답률이 같더라도 출제빈도가 높은 유형이 먼저 배치된다.

**단계 2. 프롬프트 레벨 — GPT에 전달되는 유저 프롬프트**

```javascript
const userPrompt = `수강생 ${studentName}의 개인 맞춤 학습 로드맵을 생성하세요.

수능까지 남은 기간: ${ddayCount}일 (${weekCount}주)
취약 유형 목록 (정답률 낮은 순):
${typeList}  // type_code, type_name, correct_rate, frequency_weight 포함`
```

GPT에게 "X일(Y주) 안에" 라는 시간적 제약을 명시한다. 시스템 프롬프트의 원칙 2 "현실적으로 달성 가능한 계획"과 결합되어, 남은 기간이 짧으면 GPT가 자동으로 핵심 유형에만 집중하는 계획을 생성한다.

**단계 3. 시스템 프롬프트 — "마지막 2~3주 통합 복습" 명시**

시험 직전 2~3주를 새로운 유형 학습이 아닌 통합 복습과 실전 모의고사로 배치하도록 형식을 고정한다. D-day가 임박한 수강생이 로드맵을 받을 때 마지막 주차에 불필요한 새 학습이 들어가지 않도록 방지한다.

### 4.3 JSON 출력 구조 강제

```javascript
// 반환 형식 (JSON):
{
  "summary": "전체 로드맵 요약 텍스트 (200자 이내)",
  "weeks": [
    {
      "week_number": 1,
      "items": [
        {
          "type_code": "KOR_READ_FACT",
          "type_name": "사실적 이해",
          "priority_rank": 1,
          "note": "학습 방향 및 핵심 포인트"
        }
      ]
    }
  ]
}
```

유저 프롬프트에 반환 형식을 명시하여 `response_format: { type: 'json_object' }`와 함께 파싱 오류를 원천 차단한다.

## 5. 미니 모의고사 프롬프트 (exam/system.txt)

### 5.1 실제 프롬프트

```
당신은 수능 문제 출제 전문가입니다.
수강생의 취약 유형을 집중적으로 훈련할 수 있는 개인 맞춤 미니 모의고사를 출제합니다.

출제 원칙:
1. 취약 유형 문제를 70%, 기타 유형을 30% 비율로 출제합니다
2. 실제 수능과 동일한 형식(5지선다 객관식)을 원칙으로 합니다
3. 15문항 총점 100점 기준으로 각 문항의 점수를 배분합니다
4. 문항 순서는 쉬운 것부터 어려운 순으로 배치합니다
5. 각 문항에 명확한 해설을 제공합니다

점수 배분:
- 난이도 A(하): 5점
- 난이도 B(중): 7점
- 난이도 C(상): 10점
```

### 5.2 설계 의도

**70/30 비율 명시**
시스템 프롬프트에 비율을 명시하되, 실제 비율 계산(`Math.round(15 * 0.7) = 11`)은 코드 레벨에서 수행하고 결과를 유저 프롬프트에 주입한다. GPT는 이미 계산된 숫자를 받아 문제를 구성하므로 비율 오류가 없다.

**점수 배분 명시**
문제 생성 프롬프트에는 없는 점수 체계를 모의고사 프롬프트에는 포함했다. 이는 교강사가 모의고사 결과를 점수로 평가할 수 있도록 하는 UX 요건에서 나온 결정이다.

**"쉬운 것부터" 순서 명시**
실제 수능은 쉬운 문제부터 배치되는 관행이 있다. 이를 모의고사에도 반영하여 수능 환경과의 일치성을 높이고, 수강생이 초반에 자신감을 얻고 후반 어려운 문제에 도전하도록 유도한다.

## 6. 유형 분류 프롬프트 (typeMapping/system.txt)

### 6.1 실제 프롬프트

```
당신은 수능 출제 분야 전문가입니다.
강의 내용을 분석하여 해당 강의가 다루는 수능 출제 유형을 정확히 분류합니다.

분류 원칙:
1. 강의 내용에서 명시적으로 다루는 개념과 문제 유형만 태그합니다
2. 강의 내용과 직접적으로 관련 있는 유형 코드만 선택합니다 (최소 1개, 최대 5개)
3. 유형 코드는 제공된 목록에서만 선택합니다
4. 강의 내용과 가장 관련성이 높은 순서로 배열합니다

JSON 형식으로만 응답하세요. 예: {"types": ["KOR_READ_FACT", "KOR_READ_INFERENCE"]}
```

### 6.2 설계 의도

**"명시적으로 다루는 개념만"**
강의가 미분을 설명하면 `MATH_CALC_DERIV`만 태깅하고, 관련이 있을 것 같다는 이유로 `MATH_CALC_INTEG`를 추가하지 않도록 제한. 유형 오분류는 문제 생성 단계에서 엉뚱한 유형의 문제를 생성하는 연쇄 오류로 이어지기 때문.

**최소 1개, 최대 5개 제한**
0개 반환은 파이프라인 중단, 10개 이상은 유형 희석을 유발한다. 1~5개로 제한하여 파이프라인 안정성을 확보한다.

**"제공된 목록에서만"**
유저 프롬프트에서 `suneung_types.json`의 전체 유형 코드 목록을 함께 주입한다. GPT가 목록에 없는 코드를 임의로 생성하는 것을 방지하는 closed-set 분류 전략이다.

**temperature: 0.3 (최저)**
다른 기능과 달리 유형 분류는 낮은 temperature를 사용한다. 동일한 강의를 반복 분류할 때 일관된 결과를 보장해야 하기 때문. 창의성보다 재현성이 요구되는 유일한 파이프라인 단계다.

## 7. 프롬프트 전략 종합

### temperature 선택 기준

| 기능 | temperature | 선택 이유 |
|------|-------------|-----------|
| 유형 분류 (typeMapping) | 0.3 | 동일 강의 반복 분류 시 일관성 강제 |
| 문제 생성 (questionGeneration) | 0.7 | 다양한 문제 유형·선택지 확보 |
| 로드맵 생성 (roadmap) | 0.6 | 맞춤화와 JSON 구조 안정성 균형 |
| RAG Q&A (ragQA) | 0.5 | 정확성과 자연스러운 설명 균형 |

### response_format 선택 기준

| 기능 | response_format | 이유 |
|------|-----------------|------|
| 문제 생성 | `json_object` | 구조화 파싱 필수, 스트리밍 불필요 |
| 로드맵 생성 | `json_object` | 주차별 항목 파싱 필수 |
| 유형 분류 | `json_object` | `{"types": [...]}` 단순 구조 파싱 |
| RAG Q&A | `stream: true` | 사용자 경험 — 타이핑 효과 실시간 전달 |

### 공통 패턴: "JSON 형식으로만 응답하세요"

모든 비스트리밍 프롬프트 마지막 줄에 동일한 지시어를 포함한다. `response_format: { type: 'json_object' }`와 함께 사용하면 GPT가 JSON 외 텍스트를 출력하는 것을 이중으로 방지한다.
