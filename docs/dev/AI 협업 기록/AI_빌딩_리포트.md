# CLAIQ AI 빌딩 리포트

> KIT 바이브코딩 공모전 ③번 필수 제출 서류

**서비스명**: CLAIQ (Class + AI + Q = 수능 학원 AI 플랫폼)
**팀 구성**: 2인 (기획·개발)
**배포 URL**: https://claiq.vercel.app (프론트) / https://claiq.onrender.com (백엔드)

## 1. 사용자 정의 및 문제 정의

### 설정 사용자

CLAIQ는 수능 준비 중소 학원의 4개 역할을 대상으로 설계되었다. 각 역할의 페인포인트는 네이버 카페, 에듀인뉴스, 학원 커뮤니티를 Python 크롤러로 수집한 실제 사용자 언어 기반 데이터에서 도출했다.

| 역할 | 핵심 페인포인트 |
|------|----------------|
| **교강사** | "문제 만드는 데 2~3시간" / "강의 후 개별 질문 대응 부담" / "학생 약점 파악 어려움" |
| **수강생** | "어디서부터 공부해야 할지 모름" / "질문하기 눈치 보임" / "D-day 관리 혼자 못 함" |
| **운영자** | "수강생 이탈 징조를 미리 알 수 없음" / "강사별 만족도 비교 불가" |
| **학부모** | "자녀 학습 진도를 학원에 물어봐야만 알 수 있음" |

### 구체적 문제점

크롤링 데이터가 드러낸 구조적 문제는 세 가지로 요약된다.

**문제 1. 교강사의 콘텐츠 생산 병목**
수능 유형에 맞는 5지선다 문제를 직접 출제하는 데 강의 1개당 평균 2~3시간이 소요된다. 이 시간은 강의 준비나 학생 상담에 쓰여야 할 자원이다.

**문제 2. 수강생의 자기주도 학습 공백**
"어디서부터 공부해야 하나"는 질문이 커뮤니티에서 반복된다. 오답 데이터가 있음에도 이를 분석해 개인 맞춤 계획을 제공하는 학원은 극소수다.

**문제 3. 이탈 감지의 사후성**
운영자가 학생 이탈을 인지하는 시점은 이미 등록 취소 이후인 경우가 대부분이다. 출석 데이터가 있음에도 위험 신호를 사전에 포착하는 시스템이 없다.

## 2. 솔루션 핵심 기능 및 차별점

### 핵심 기능 (5개 AI 파이프라인)

**파이프라인 1. 강의 → 문제 자동 생성**
교강사가 강의 음성을 업로드하면 Whisper STT → 텍스트 청킹 → pgvector 임베딩 → GPT-4o 문제 생성까지 자동 처리된다. 생성된 문제는 교강사가 승인/수정/반려하는 Human-in-the-Loop 검수를 거쳐 공개된다.

**파이프라인 2. RAG 기반 AI Q&A**
수강생의 질문을 임베딩하여 강의 청크 중 유사도 상위 5개를 검색(cosine distance), GPT가 해당 컨텍스트만 참조하여 답변한다. 강의 범위 이탈 질문은 교강사에게 자동 에스컬레이션된다.

**파이프라인 3. 수능 D-day 역산 학습 로드맵**
수강생의 유형별 정답률과 수능 출제 빈도 가중치를 결합한 `priority_score`를 계산하고, D-day까지 남은 주수에 맞춰 주차별 학습 계획을 GPT가 생성한다.

**파이프라인 4. 약점 기반 미니 모의고사**
약점 유형 70% + 기타 유형 30% 비율로 15문항을 자동 구성한다. 풀이 결과는 즉시 `student_type_stats`에 반영되어 로드맵과 연동된다.

**파이프라인 5. 이탈 예측 대시보드**
3일 미출석 시 `at_risk`, 7일 미출석 시 `inactive`로 분류하는 규칙 기반 이탈 감지가 매일 오전 9시 크론잡으로 실행된다. 운영자는 Solapi SMS로 위험 학생에게 개입할 수 있다.

### 차별점

| 항목 | 기존 학원 관리 시스템 | CLAIQ |
|------|----------------------|-------|
| 문제 생성 | 교강사 수동 출제 (2~3시간) | Whisper + GPT 자동 생성 (5분 이내) |
| 학생 질문 대응 | 카카오톡·전화 대기 | 24시간 RAG Q&A + 에스컬레이션 |
| 학습 계획 | 일괄 커리큘럼 | D-day·약점 기반 개인 맞춤 로드맵 |
| 이탈 감지 | 등록 취소 후 인지 | 출석 데이터 기반 사전 감지 |
| 학부모 소통 | 전화 문의 | SMS 자동 발송 |

## 3. 도입 시 기대 개선점

### 정성적 효과

**교강사**
- 강의 업로드 후 문제 검수·승인만으로 콘텐츠 생산 완결. 문제 출제 시간을 강의 준비와 학생 상담에 재투자 가능.
- 에스컬레이션 목록을 통해 학생 질문 유형을 파악하고 다음 강의 개선에 활용.

**수강생**
- 수능까지 남은 기간과 내 약점을 반영한 주차별 학습 계획으로 "어디서부터"라는 막막함 해소.
- 심야나 주말에도 AI Q&A로 즉각 학습 지원. 질문 눈치 부담 없음.

**운영자**
- 이탈 위험 학생을 3일 전에 감지하여 SMS 개입. 이탈 예방을 통한 수강 유지율 향상.
- 강의별 문제 정답률과 학생 통계로 강사별 교육 효과 비교 가능.

**학부모**
- 자녀의 출석, 정답률, 로드맵 달성 현황을 학원 측 SMS로 주기적으로 수신.

### 정량적 기대치 (추정)

| 지표 | 추정 근거 |
|------|-----------|
| 문제 출제 시간 90% 단축 | 수동 2~3시간 → AI + 검수 15~20분 |
| 강의당 AI 처리 비용 약 30원 | STT(25원) + 임베딩(1원) + 문제생성(4원), OpenAI 공식 요금 기준 |
| Q&A 응답 즉시화 | 카카오톡 평균 응답 대기 수시간 → AI 즉시 + 에스컬레이션 24시간 내 |

## 4. AI 도구 및 모델 선택 이유

### Claude Code 선택 이유

Claude Code는 단순 코드 자동완성 도구가 아닌 **에이전트 오케스트레이션 플랫폼**으로 활용했다.

**CLAUDE.md 컨텍스트 유지**: 프로젝트 기술 스택, 개발 규칙, AI 협업 로깅 형식을 `CLAUDE.md`에 정의했다. 매 세션에서 이 컨텍스트가 자동 로드되어 에이전트 간 일관성을 보장했다.

**에이전트 병렬 실행**: 독립적인 작업(프론트엔드 구현 / 백엔드 구현 / DB 스키마 설계)을 동시에 다른 에이전트에 위임하여 개발 기간을 단축했다. 예를 들어 초기 설계 단계에서 `architect`, `database-reviewer`, `security-reviewer` 3개 에이전트를 병렬 실행했다.

**오케스트레이터-워커 분리**: 메인 Claude(오케스트레이터)는 코드를 직접 작성하지 않고 전체 흐름만 설계한다. 실제 코드 변경은 각 전문 에이전트(`express-engineer`, `react-specialist`)에 위임하여 역할 경계를 명확히 했다.

**MCP 통합**: Context7 MCP를 통해 `@google/genai`, `pg`, `bullmq` 등 외부 라이브러리의 최신 API 문서를 세션 중 즉시 조회, 잘못된 API 호출 오류를 사전에 방지했다.

### AI Tool Stack (확인된 구현)

**Claude (Anthropic)**
전체 설계·코딩·문서화 파트너로 활용. 플랫폼 아키텍처 설계, 백엔드 API 개발, 프론트엔드 컴포넌트 구현, 데이터베이스 스키마 설계, 테스트 자동화(Playwright)를 담당했다.

**GPT-4o** (비용 최적화 모델)
- 문제 생성: `questionGenerator.js`에서 `generateQuestions()` 호출 시 사용. temperature=0.7로 다양한 문제 유형 확보.
- Q&A 응답: `ragQA.js`에서 스트리밍 응답 시 temperature=0.5로 정확성과 자연스러움 균형.
- 로드맵 생성: `roadmapGenerator.js`에서 D-day 역산 계획 생성 시 temperature=0.6.
- 미니 모의고사: `examGenerator.js`에서 약점 70% + 기타 30% 비율 구성.
- 유형 분류: `typeMapper.js`에서 강의 내용 기반 수능 유형 코드 자동 분류, temperature=0.3 (일관된 분류 강제).
환경변수 `OPENAI_MODEL_CHAT`으로 gpt-4o 등으로 즉시 교체 가능. (테스트 및 로컬 환경에서는 mini로 테스트 진행)

**Whisper-1** (STT, `back/src/ai/whisper.js`)
강의 음성 파일(MP3/MP4)을 텍스트로 변환. 한국어 설정(`language: 'ko'`), 25MB 파일 크기 제한. 임시 파일로 처리 후 즉시 삭제.

**text-embedding-3-small** (임베딩, `back/src/ai/embedding.js`)
강의 청크를 1536차원 벡터로 변환하여 pgvector에 저장.
배치 처리: 20개씩 묶어서 처리(BATCH_SIZE=20), 배치 간 50ms 딜레이(BATCH_DELAY_MS=50).
rate limit 대응: 429/503 오류 시 exponential backoff 재시도 (1s → 2s → 4s, 최대 3회).

| 모델 | 용도 | 환경변수 | 기본값 |
|------|------|---------|--------|
| gpt-4o | Chat (문제/QA/로드맵/모의고사) | `OPENAI_MODEL_CHAT` | gpt-4o |
| Whisper-1 | STT | `OPENAI_MODEL_STT` | whisper-1 |
| text-embedding-3-small | 임베딩 | `OPENAI_MODEL_EMBEDDING` | text-embedding-3-small |

## 5. 도구별 AI 활용 전략

### Claude Code 에이전트 분업 전략

CLAIQ 개발에 투입된 10개 전문 에이전트와 각각의 역할:

| 에이전트 | 담당 도메인 | 주요 산출물 |
|----------|-------------|-------------|
| `planner` | 기능 설계 전 계획 수립 | 강의 업로드 파이프라인 5단계 구현 계획 |
| `architect` | 시스템 아키텍처 | AI 파이프라인 레이어 분리, Human-in-the-Loop 설계 |
| `express-engineer` | 백엔드 전체 | 12개 도메인 API, AI 모듈 7개, SSE 실시간 처리 |
| `react-specialist` | 프론트엔드 전체 | 31개 페이지, 35개 컴포넌트, Zustand v5 스토어 |
| `database-reviewer` | PostgreSQL + pgvector | 27개 테이블, IVFFlat 인덱스, 소프트삭제 패턴 |
| `security-reviewer` | 보안 감사 | RBAC 설계 검증, API 키 관리, 파일 업로드 검증 |
| `build-error-resolver` | 빌드 오류 | pgvector 의존성, OpenAI SDK 호환성 문제 해결 |
| `tdd-guide` | 품질 검증 | 핵심 파이프라인 로직 테스트 전략 |
| `refactor-cleaner` | 코드 품질 | 중복 제거, 함수 분리, 800줄 이하 파일 유지 |
| `doc-updater` | 문서 | API 문서, ADR 3개, AI 파이프라인 문서 |

**병렬 실행 사례:**

```
초기 설계 단계 — 3개 에이전트 동시 실행:
├── architect        → 전체 시스템 아키텍처 설계
├── database-reviewer → 데이터베이스 스키마 초안 작성
└── security-reviewer → 보안 요구사항 사전 정의

구현 완료 후 검증 — 3개 에이전트 동시 실행:
├── security-reviewer → 보안 취약점 스캔
├── tdd-guide         → 테스트 커버리지 확인
└── refactor-cleaner  → 코드 품질 검토

문서화 단계 — 3개 에이전트 동시 실행:
├── doc-updater (AI 파이프라인 문서)
├── doc-updater (API 엔드포인트 문서)
└── doc-updater (ADR 문서 3개)
```

### 프롬프트 관리 전략

모든 시스템 프롬프트는 코드와 분리된 외부 파일로 관리된다.

```
back/src/prompts/
├── questionGeneration/
│   ├── korean.txt   # 국어 문제 출제 전문가 역할
│   ├── math.txt     # 수학 문제 출제 전문가 역할
│   ├── english.txt  # 영어 문제 출제 전문가 역할
│   └── system.txt   # 과목 불명 폴백
├── ragQA/
│   └── system.txt   # AI 튜터 역할 + hallucination 방지
├── roadmap/
│   └── system.txt   # 수능 학습 전략가 역할
├── exam/
│   └── system.txt   # 미니 모의고사 출제 전문가 역할
└── typeMapping/
    └── system.txt   # 수능 유형 분류 전문가 역할
```

이 구조의 장점:
- 프롬프트 개선 시 코드 배포 없이 파일만 교체 가능
- 과목별 특성을 각 파일에 독립적으로 반영
- `readFileSync`로 서버 시작 시 로드, 요청마다 파일 I/O 없음

> **과목 확장성**: MVP는 국어·수학·영어 3과목 중심이나, DB에는 수능 전 과목(국어 독서·문학·언어와 매체·화법과 작문 / 수학 I·II·미적분·확통·기하 / 영어 독해·듣기 / 한국사 / 사회탐구 2종 / 과학탐구 4종 등 총 18개 과목)이 사전 설계되어 있어 즉시 활성화 가능. 새 과목 추가 시 DB INSERT 1건과 프롬프트 파일 추가만으로 확장됨.

### 데이터 흐름 전체 요약

```
[교강사] 강의 파일 업로드 (MP3/MP4/PDF)
    │
    ▼
Whisper-1 STT → 강의 스크립트 생성
    │
    ▼
텍스트 청킹 (500토큰, 50토큰 오버랩)
    │
    ▼
text-embedding-3-small → 1536차원 벡터
    │
    ▼
pgvector 저장 (lecture_chunks 테이블)
    │
    ├──→ GPT-4o 문제 생성 → [교강사 검수] → 공개
    │
[수강생] 문제 풀기 → student_type_stats 갱신
    │
    ├──→ AI Q&A: 질문 임베딩 → 유사도 검색(top-k=5) → GPT 컨텍스트 주입 → 스트리밍 답변
    │
    ├──→ 학습 로드맵: 약점 분석 → D-day 역산 → GPT 주차별 계획
    │
    └──→ 미니 모의고사: 약점 70% + 기타 30% 자동 구성
```

## 6. 토큰 효율·유지보수성·재현성

### 토큰 최적화 전략

**gpt-4o 기본값**
모든 Chat API 호출에 `env.openai.modelChat`(기본값: `gpt-4o`)을 사용한다. 환경변수 `OPENAI_MODEL_CHAT`만 변경하면 gpt-4o로 즉시 전환 가능.

**RAG로 컨텍스트 최소화**
강의 전체 텍스트를 프롬프트에 주입하는 대신, 질문과 유사도가 높은 청크 5개(top-k=5)만 컨텍스트로 주입한다. 전체 강의 대비 약 90% 토큰 절감.

**임베딩 배치 처리**
단건 API 호출 대신 20개씩 배치(`BATCH_SIZE=20`)로 처리, 배치 간 50ms 딜레이로 rate limit 방지. 같은 강의에 대한 중복 임베딩은 청크 단위 캐시로 방지.

**temperature 최적화**
유형 분류(typeMapping): `temperature: 0.3` — 일관된 분류 강제
문제 생성: `temperature: 0.7` — 다양한 문제 유형 확보
RAG Q&A: `temperature: 0.5` — 정확성과 자연스러운 답변 균형
로드맵: `temperature: 0.6` — 맞춤화와 일관성 균형

**월 예상 비용 (추정, OpenAI 공식 요금 기준)**

| 항목 | 예상 비용 |
|------|-----------|
| Whisper STT (강의 100시간 기준) | 약 3,600원 |
| 임베딩 생성 (강의 100개 기준) | 약 1,000원 |
| 문제 생성 (gpt-4o, 강의 100개 × 20문제) | 약 2,000원 |
| Q&A 응답 (gpt-4o, 월 1,000건) | 약 3,000원 |
| **합계** | **약 1~2만원/월** |

### 유지보수성

**환경변수 기반 모델 전환**
`env.js`에서 OpenAI 모델명을 환경변수로 관리한다.
```
OPENAI_MODEL_CHAT=gpt-4o    # Chat 모델 (문제생성/Q&A/로드맵)
OPENAI_MODEL_EMBEDDING=text-embedding-3-small  # 임베딩 모델
OPENAI_MODEL_STT=whisper-1       # STT 모델
```
모델 업그레이드 시 코드 변경 없이 환경변수만 수정.

**청크 파라미터 조정**
```
RAG_CHUNK_SIZE=500       # 청크 토큰 크기
RAG_CHUNK_OVERLAP=50     # 오버랩 토큰 수
RAG_TOP_K=5              # 유사도 검색 결과 수
```
RAG 정확도 튜닝이 필요할 때 코드 변경 없이 환경변수로 즉시 조정.

**이탈 감지 기준 조정**
```
CHURN_RISK_DAYS=3        # at_risk 분류 기준일
CHURN_INACTIVE_DAYS=7    # inactive 분류 기준일
```
학원 운영 특성에 맞게 파라미터 조정 가능.

**프롬프트 파일 독립 관리**
`back/src/prompts/` 하위 `.txt` 파일을 교체하면 서버 재시작만으로 프롬프트 반영. 비개발자도 프롬프트 개선에 참여 가능한 구조.

### 재현성

**CLAUDE.md — AI 협업 환경 재현**
`CLAUDE.md`에는 프로젝트 기술 스택, 개발 규칙, 에이전트 활용 원칙이 명시되어 있다. 새 Claude Code 세션을 시작할 때 이 파일이 자동 로드되어 동일한 개발 환경을 즉시 재현한다.

**AGENTS.md — 에이전트 협업 구조 문서화**

기획 리서치부터 배포까지 8개 주요 작업의 결정 근거가 기록되어 있다. "왜 gpt-4o인가", "왜 RAG인가", "왜 Vercel+Render인가" 등 모든 핵심 결정이 재현 가능한 형태로 남아 있다.

**환경변수 기반 설정**
시크릿을 제외한 모든 AI 파라미터(모델명, 청크 크기, top-k, 이탈 기준일)가 환경변수로 외부화되어 있다. `.env` 파일 하나로 동일한 동작을 다른 환경에서 재현 가능.
