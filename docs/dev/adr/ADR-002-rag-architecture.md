# ADR-002: RAG Q&A 아키텍처

## 상태

**승인됨** - 2026-04-07

---

## 맥락

수강생이 강의 내용에 대해 질문했을 때 AI가 정확하게 답변할 수 있는 방법을 결정해야 했습니다. 단순히 GPT에게 질문을 전달하면 강의와 무관한 답변이나 환각(hallucination)이 발생할 수 있어, 강의 내용에 기반한 신뢰할 수 있는 Q&A 시스템이 필요했습니다.

### 검토된 선택지

#### 선택지 1: Fine-tuning

강의 텍스트로 GPT 모델을 파인튜닝하여 전용 모델을 만드는 방식.

**장점:**
- 강의 내용을 모델이 내재화
- 빠른 응답 속도

**단점:**
- 강의가 업데이트될 때마다 재학습 필요 (비용 및 시간 소요)
- 파인튜닝 비용: 강의 1개당 수만 원 예상
- 환각 완전 제거 불가
- 강의별 모델 관리 복잡성 급증

#### 선택지 2: Prompt Engineering만 사용

GPT 컨텍스트 창에 강의 전체를 넣고 질문하는 방식.

**장점:**
- 구현 단순

**단점:**
- 강의가 길면(2시간+) 컨텍스트 초과
- 긴 컨텍스트 = 높은 비용
- 관련 없는 내용까지 포함되어 응답 품질 저하

#### 선택지 3: pgvector 기반 RAG (Retrieval-Augmented Generation)

강의를 청크로 분할하고 벡터 DB에 저장, 질문과 유사한 청크만 검색하여 컨텍스트로 제공하는 방식.

**장점:**
- 강의 업데이트 시 해당 청크만 재임베딩 (부분 업데이트 가능)
- 관련 내용만 컨텍스트에 포함 → 비용 최소화
- 환각 가능성 최소화 (강의 기반 답변 강제)
- pgvector를 기존 PostgreSQL에 확장으로 추가 → 별도 인프라 불필요

**단점:**
- 임베딩 생성 및 검색 로직 구현 필요
- 청킹 전략에 따라 검색 품질이 달라짐

---

## 결정

**pgvector 기반 RAG 아키텍처를 채택**

구체적인 구현 방식:
- 벡터 DB: PostgreSQL + pgvector 확장 (별도 Pinecone/Weaviate 미사용)
- 임베딩 모델: text-embedding-3-small (1536차원)
- 청킹: 500토큰, 50토큰 오버랩
- 검색: cosine similarity, top-k=5
- 생성: gpt-4o-mini 스트리밍
- 에스컬레이션: 관련 청크가 없거나 AI 응답에 "교강사에게 문의" 또는 "강의 범위를 벗어" 포함 시 교강사에게 에스컬레이션

---

## 근거

### 1. 강의별 최신 내용 즉시 반영

교강사가 강의를 수정하거나 보충 자료를 추가할 때, Fine-tuning이라면 전체 재학습이 필요하지만 RAG는 해당 청크만 재임베딩하면 됩니다.

```
Fine-tuning 업데이트: 강의 재학습 → 수 시간 ~ 수일, 수만 원
RAG 업데이트: 변경된 청크 재임베딩 → 수 초, 수 원
```

이는 "살아있는 강의" 컨셉에 부합합니다. 교강사가 보충 설명을 추가하면 즉시 Q&A에 반영됩니다.

### 2. 환각(Hallucination) 최소화

시스템 프롬프트에서 강의 컨텍스트 외 정보 사용을 명시적으로 금지합니다.

```
"아래 강의 내용만을 근거로 답변하세요.
 근거가 없으면 '강의 내용에서 찾을 수 없습니다'라고 답하세요."
```

검색된 top-5 청크에 없는 내용은 AI가 답변하지 않고 교강사에게 에스컬레이션합니다. 이는 교육 플랫폼에서 잘못된 정보 전달을 방지하는 핵심 장치입니다.

### 3. 비용 효율

Fine-tuning 대비 비용 비교:

| 시나리오 | Fine-tuning | RAG |
|----------|-------------|-----|
| 초기 설정 | 강의당 수만 원 | 강의당 수십 원 (임베딩만) |
| Q&A 1회 | 모델 호출 비용 | top-5 청크 + gpt-4o-mini |
| 강의 업데이트 | 재학습 필요 | 변경 청크만 재임베딩 |
| **월 100개 강의** | **수백만 원** | **수천 원** |

### 4. 기존 인프라 활용 (pgvector)

별도의 벡터 DB 서비스(Pinecone, Weaviate, Milvus 등)를 도입하지 않고 기존 PostgreSQL에 pgvector 확장을 추가합니다.

**이점:**
- 추가 인프라 비용 없음
- 관계형 데이터(강의 메타데이터)와 벡터 데이터를 단일 DB에서 JOIN 가능
- PostgreSQL의 트랜잭션, 백업, 보안 정책 그대로 적용

```sql
-- 벡터 유사도 검색 (실제 구현 기준)
-- pgvector 제약으로 임베딩 벡터는 '[...]'::vector 형식으로 직접 삽입
SELECT lc.id, lc.lecture_id, lc.content,
       1 - (lc.embedding <=> '[...]'::vector) AS similarity
FROM lecture_chunks lc
WHERE lc.teacher_id = $1
  AND lc.academy_id = $2
  AND lc.embedding IS NOT NULL
ORDER BY lc.embedding <=> '[...]'::vector
LIMIT $3;
```

## 결과

### 구현된 Q&A 흐름

```
수강생 질문 입력
       │
       ▼
질문 텍스트 임베딩 생성 (text-embedding-3-small)
       │
       ▼
pgvector cosine similarity 검색 (top-5 청크)
       │
관련 청크 존재?
 ┌─────┴─────┐
YES          NO
 │            │
 ▼            ▼
GPT-4o-mini  에스컬레이션
스트리밍     (교강사 알림)
응답
 │
 ▼
수강생에게 스트리밍 전송
```

### 구현 상태
- 강의 관련 질문에 대해 관련 청크가 존재할 경우 컨텍스트 기반 답변을 제공하도록 구현됨
- 스트리밍 응답으로 체감 지연 최소화 (첫 토큰까지 대기 없이 텍스트 순차 출력)
- 에스컬레이션: 관련 청크가 없거나 AI 응답이 강의 범위를 벗어날 경우 교강사에게 알림

### 트레이드오프 인정
- 청킹 경계에서 문맥이 잘릴 수 있음 → 50토큰 오버랩으로 완화
- 에스컬레이션 민감도는 운영 중 피드백을 통해 조정 예정

### 향후 개선 방향
- HyDE(Hypothetical Document Embeddings): 질문을 가상 답변으로 확장 후 검색하는 방식 검토
- Reranking: top-5 검색 후 크로스 인코더로 재순위화
- 피드백 루프: 수강생의 "도움됨/안됨" 피드백으로 검색 품질 지속 개선
